## Task 1: Fix Cost-Tallying Bug

**File:** `scripts/full_L3_simulation.py`

**Problem:** When `--service-level` was set, the `costs` object was replaced with adjusted values (e.g., at SL=0.50, cu became 0.2 instead of 1.0). This same object was passed to `simulate_week()` for realized cost tallying, meaning shortage costs were evaluated at 1/5th their true value at lower service levels. All prior backtest results were invalid — lower SLs appeared artificially cheap.

**Fix:**
- Added `eval_costs` parameter (always cu=1.0, co=0.2) to `run_full_simulation()`
- `simulate_week()` now uses `eval_costs` for tallying, while `_order_costs()` uses adjusted costs for order generation only
- The old line `costs = costs_for_service_level(...)` in `main()` was removed; replaced with `order_costs` for display only
- Added per-SKU-per-week diagnostic output (`sku_detail.parquet`) from each simulation run
- Added `--static-folds` flag for train-once vs sequential comparison

**Impact:** With corrected costs, SL=0.833 is optimal for SLURP models (not SL=0.50 as previously reported).

## Task 5: Weighted Pinball at All Quantiles

**Script:** `scripts/compute_pinball_all_quantiles.py` (new)

Computed pinball loss at 14 quantile levels (13 model quantiles + SL=0.833) for all models, in three variants:
1. **Unweighted:** standard quantile loss
2. **Cost-weighted:** cu * undershoot + co * overshoot (asymmetric business costs)
3. **Volume-weighted:** unweighted × (SKU's total 8-week demand / mean total demand)

**Key Findings:**
- SLURP models dominate at all quantile levels
- Cost-weighted pinball is minimized near q=0.80-0.90 for SLURP (0.826-0.825)
- LightGBM's cost-weighted pinball is U-shaped, lowest at q=0.80 (0.836)
- Seasonal naive is ~40x worse at every quantile
- Volume-weighted pinball amplifies SLURP's advantage on high-movers

**Output:** `reports/pinball/` — pinball_summary.csv, pinball_cost_weighted.csv, pinball_volume_weighted.csv

## Task 7: Hypothesis Analysis

**Script:** `scripts/analyze_hypotheses.py` (new)

Synthesized all results into a structured analysis of the four paper hypotheses.

### H1 (Jensen Gap): Confirmed with caveat
- Positive for SLURP: +€815 (bootstrap), +€544 (stockout_aware)
- Negative for LightGBM (-€827) and seasonal_naive (-€1,026)

- bootstrap wins at SL=0.833 (by €95)
- Better calibration at 0.80 quantile (83.5% vs 82.0%) confirms wider bands
- Trade-off depends on risk appetite

### H3 (SURD): Deferred
- SURD models not trained this round
- SURD transforms computed; show variance reduction for ~40% of series

### H4 (Sequential Consistency): Supported
- Rank reversal between pinball (stockout_aware #1) and simulation (bootstrap #1)
- Train-once vs sequential also shows model-dependent effects invisible to period metrics

## Task 9: Safety Stock Formula Comparison (Workstream 3)

**Script:** `scripts/backtest_safety_stock_policies.py` (new)

Implemented 6 ordering policies sharing the same 8-week simulation engine (identical initial state, actuals, L=3 mechanics):

1. z·σ·√L — CSL-based, ignores review period
2. z·σ·√(L+R) — accounts for weekly review
3. z·RMSE·√(L+R) — uses forecast error instead of demand variability
4. k·RMSE — optimize k via sweep (similar to VN2 winner's approach)
5. k·MAE — more stable error indicator
6. Density SIP — our full probabilistic approach

**Sweep:** z ∈ {0.50, 0.84, 1.00, 1.28, 1.64, 2.00} for policies 1-3; k ∈ {0.5, 1.0, ..., 5.0} for policies 4-5. Per-SKU σ, RMSE, MAE computed from pre-competition demand and slurp_bootstrap forecast errors.

**Key Results:**

| Policy | Best Param | Total | Fill Rate |
|--------|----------:|------:|----------:|
| z·σ·√L | z=0.50 | €6,498 | 88.3% |
| z·σ·√(L+R) | z=0.50 | €6,618 | 89.0% |
| z·RMSE·√(L+R) | z=0.50 | €6,018 | 89.8% |
| k·RMSE | k=0.50 | €5,807 | 87.1% |
| k·MAE | k=0.50 | €5,771 | 86.4% |
| **Density SIP** | **SL=0.833** | **€5,169** | **79.9%** |

**Finding:** Density SIP beats all classical formulas by €602-€1,449, confirming H1 (Jensen Gap). Even k·MAE (€5,771) doesn't beat the benchmark (€5,248), while SIP does.

**Output:** `reports/safety_stock/policy_comparison.csv`, `policy_comparison.md`
